# Copilot Instructions – eShopAcademy

## Project Goal
eShopAcademy is a .NET 8 microservices-based e-commerce platform focused on
real-world architecture, correctness, and maintainability.

This is NOT a toy project.
Architectural clarity is prioritized over shortcuts.
Review ARCHITECTURE.md for detailed guidelines.

---

## Architecture
- Microservices with clear service boundaries
- Clean Architecture / DDD-inspired layering
- Each service owns its data and persistence
- No shared databases between services

---

## Communication
### Synchronous
- gRPC for internal service-to-service calls

### Asynchronous
- RabbitMQ / Azure Service Bus for integration events
- Messaging is preferred over direct calls when feasible
- Eventual consistency is expected and accepted

---

## Workflow & Orchestration
- Saga pattern using MassTransit
- Order Service orchestrates most business workflows
- Sagas react to domain and integration events
- Avoid tight coupling between services

---

## Payments
- Strategy pattern for PSP integrations
- Providers implemented as adapters
- Payment logic must remain isolated and extensible

---

## Local Development
- .NET Aspire is used for local orchestration
- All services must be runnable locally
- Azure Service Bus will be used in production

---

## Design Rules (DO / DO NOT)

### DO
- Follow existing abstractions and naming conventions
- Modify multiple files when required to implement a feature correctly
- Prefer events over direct dependencies
- Keep infrastructure concerns isolated
- Add all entities and contracts to the corresponding domain projects and then push the nugets and upgrade the consumers
- Create entity framework migrations whenever the model changes by running the dotnet ef command, not manually creating the script files 

### DO NOT
- Collapse layers for simplicity
- Introduce new architectural styles without justification
- Replace messaging with synchronous calls
- Introduce shortcuts that reduce clarity

---

## Expected Copilot Behavior
- Respect existing architecture
- Implement features across services when needed
- Ask for clarification only when a design decision is ambiguous
